# Bug Scan — 2026-04-10

Targeted scan of files touched in commit `6125b2a` (goal-coach phase 1-5, UI redesign, retry helper).

**Totals:** 3 Critical · 3 High · 3 Medium · 3 Low

## Critical

### BUG-C1 — NaN limit param in tags API
**File:** `src/app/api/tags/route.ts:11`

`parseInt(url.searchParams.get("limit") ?? "50")` returns `NaN` for non-numeric input like `?limit=abc`. Prisma silently ignores NaN in `take:`, so the query returns ALL tags instead of the 50 cap. This can leak the full tag table to any caller.

**Fix:** `const limit = Math.min(Math.max(parseInt(raw, 10) || 50, 1), 1000);`

---

### BUG-C2 — Anthropic response content not length-checked
**Files:** `src/app/api/coach/route.ts:124`, `src/lib/usda.ts:53`

Both access `response.content[0]` without checking `response.content.length > 0`. If Anthropic returns an empty content array (rare but possible with tool_use blocks or safety filtering), `assistantText` silently becomes `""` and gets persisted as a valid "blank" assistant message. The coach appears to have answered but said nothing.

**Fix:** Add `if (!response.content.length) throw new Error("empty response");` before indexing, or filter for `type === "text"` first.

---

### BUG-C3 — Race condition on isPrimary cascade
**File:** `src/app/api/goals/route.ts:36-40`

When creating a goal with `isPrimary: true`, the code first unsets all other primary goals, then creates the new one. Two concurrent POST requests both setting `isPrimary: true` can interleave such that the final DB state has multiple primaries, or worse: the second request's unset races the first's create and leaves zero primaries.

**Fix:** Wrap in `prisma.$transaction([...])`. Same pattern needs to apply to PATCH in `[id]/route.ts`.

## High

### BUG-H1 — Goal status not validated in PATCH
**File:** `src/app/api/goals/[id]/route.ts:14-31`

The PATCH handler accepts any `status` string and saves it directly. Invalid values like `"pending"`, `"on_hold"`, or random strings will land in the DB, breaking the active/archived filters used by the coach context builder and UI.

**Fix:** `if (body.status && !["active","archived","completed","abandoned"].includes(body.status)) return 400;`

---

### BUG-H2 — `focusGoal` null not handled in prompt section
**File:** `src/app/api/coach/route.ts:88-92`

`goalSystemPromptSection(focusGoal)` is called with a possibly-null value. If no primary goal exists and no `focusGoalId` is provided, the function receives null. Depending on its implementation, this either produces a malformed system prompt or throws. Verify the function signature and add a null guard at the call site if needed.

**Fix:** `const section = focusGoal ? goalSystemPromptSection(focusGoal) : "";`

---

### BUG-H3 — Unbounded `parseInt` without radix
**File:** `src/app/api/tags/route.ts:11`

Related to BUG-C1. Even after clamping, `parseInt` should always be called with radix `10` to avoid surprising octal behavior on legacy inputs.

**Fix:** `parseInt(raw, 10)` (part of the BUG-C1 fix).

## Medium

### BUG-M1 — Optimistic isPrimary update not fully reverted on failure
**File:** `src/components/goals/goals-manager.tsx:177-203`

When toggling `isPrimary`, the client optimistically flips the flag and updates sibling goals locally. On API error, the catch only surfaces a toast — the sibling cascade is not undone, so the UI diverges from the server.

**Fix:** Snapshot `goals` before the optimistic mutation and restore it in the catch block.

---

### BUG-M2 — No auth on goals/coach/tradeoffs endpoints
**Files:** `src/app/api/coach/**`, `src/app/api/goals/**`, `src/app/api/workouts/[id]/goals/route.ts`

None of the new endpoints check auth. Fine for a single-user local dev app, but explicitly note it as blocking any deploy or multi-user testing. Add to the "Known Limitations" section of the PRD so it doesn't surprise future-you.

**Fix:** Out of scope for solo dev, but document. If shipping: add middleware that injects `userId` and scopes every `where:` clause.

---

### BUG-M3 — `deleteSession` fetch not awaited
**File:** `src/components/coach/chat-interface.tsx:233-237`

The session-delete handler removes the session from local state but fires the DELETE fetch without `await` or error handling. If the request fails, the UI shows the session gone but it's still in the DB.

**Fix:** `const res = await fetch(...); if (!res.ok) { restoreState(); showError(); }`.

## Low

### BUG-L1 — `estimateMacros` swallows parse failures
**File:** `src/lib/usda.ts:59, 68, 82`

On JSON parse failure, the function returns `[]` and logs. The nutrition POST handler then happily records "no food entries" and returns success, so the user sees their meal logged with zero macros and no error.

**Fix:** Throw a `MacroParseError` and surface it to the client as a toast.

---

### BUG-L2 — Chat history fetch unbounded
**File:** `src/app/api/coach/route.ts:65-69`

The session include loads `messages` without `take:`. A chat session with thousands of messages will pull them all into memory every turn.

**Fix:** `messages: { take: 200, orderBy: { createdAt: "desc" } }` then reverse for history.

---

### BUG-L3 — Context cache not invalidated on goal change
**File:** `src/app/api/coach/route.ts:29-39`

The 5-minute context cache is keyed by `focusGoalId`. If the user renames a goal, changes its subtype, or creates a new tradeoff-triggering log, the next coach message up to 5 minutes later will see stale context.

**Fix:** Expose a `cachedContext = null` reset from mutation handlers, or shorten TTL to 60s.
