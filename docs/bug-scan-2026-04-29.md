## High

### BUG-H1 — `Promise.all` on multi-source reads has no partial-fail boundary
**Files:**
- `src/app/api/hyrox/today/route.ts:58-93`
- `src/app/body/hyrox/page.tsx:79-109`

Both call sites fan out 5 independent Prisma queries (readiness, sleep, recent sleeps, cycle phase, last hard session) inside `Promise.all`. If any single query rejects (timeout, transient connection drop), the whole endpoint 500s. The downstream `recommendSession()` is explicitly written to accept `null` for every signal — partial-fail should degrade to a less-informed recommendation, not a hard error. This is the BUG-006 pattern that `buildCoachContext()` already fixed via `Promise.allSettled` + `val()`.

**Fix:** Switch both sites to `Promise.allSettled` and extract values defensively (treat any rejection as `null`). Pattern is already established in `src/lib/coach-context.ts`.

### BUG-H2 — `goal.status` still accepts arbitrary strings in PATCH (carry-over from 2026-04-10)
**File:** `src/app/api/goals/[id]/route.ts:15-17, 26-28`

The 2026-04-10 scan's BUG-H1 flagged this and was never closed — the `a69847c` "fix 3 critical bugs" commit only touched BUG-C1/C2/C3. The PATCH still does `for (const field of [..., "status", ...])` and writes whatever the client sends. The only filtering is `data.status === "completed" → "archived"`; values like `"pending"`, `"on_hold"`, or random strings still land in the DB and break the active/archived filters used by Hyrox plan auto-archival, the coach context builder, and the goal lens lookup.

The risk is now higher than in the prior scan because `syncHyroxPlanForGoal` runs in the same transaction — a malformed status flowing through PATCH could orphan the plan/goal status pair.

**Fix:** Add `if (body.status && !["active","archived","completed","abandoned"].includes(body.status)) return 400` before the field copy loop.

### BUG-H3 — `currentBlock` field accepts any string in Hyrox plan PATCH
**File:** `src/app/api/hyrox/plan/[id]/route.ts:105-107`

```ts
if (body.currentBlock !== undefined) {
  data.currentBlock = body.currentBlock;
}
```

No enum validation. The Block type is `accumulation | transmutation | realization | taper | complete`. Anything else stored here is read back by the recommender's `WEEKDAY_TEMPLATE[blockState.block]` lookup — except the lookup uses `currentBlock()` (computed from dates), so the persisted field is mostly cosmetic. Still, a typo or malicious value sticks until the next plan recalc and shows up in the UI block badge. Same family of validation gaps as BUG-H2.

**Fix:** `if (body.currentBlock && !["accumulation","transmutation","realization","taper","complete"].includes(body.currentBlock)) return 400`.

## Medium

### BUG-M1 — Date strings constructed without validation in Hyrox plan PATCH
**File:** `src/app/api/hyrox/plan/[id]/route.ts:90-95`

```ts
if (body.raceDate !== undefined) data.raceDate = new Date(body.raceDate);
if (body.startDate !== undefined) data.startDate = new Date(body.startDate);
```

`new Date("garbage")` returns `Invalid Date`. Prisma will reject it, but with a generic 500 rather than a clear 400 with a usable message. Worse, if the raw value is a partial ISO that JS happily parses to a valid-but-wrong date (`new Date("2026-13")` → Jan 2027), it lands silently.

**Fix:** Wrap each in a parse guard: `const d = new Date(raw); if (Number.isNaN(d.getTime())) return 400`.

### BUG-M2 — Same date-string issue in goal create/update
**Files:**
- `src/app/api/goals/route.ts:57` — `deadline ? new Date(deadline) : null`
- `src/app/api/goals/[id]/route.ts:24` — `body.deadline ? new Date(body.deadline) : null`

Same as BUG-M1. Now load-bearing because `syncHyroxPlanForGoal` reads `goal.deadline` as the `raceDate` for the auto-created plan — an Invalid Date silently propagates into HyroxPlan, breaking `weeksToRace()` (returns NaN, `autoScaleBlocks` returns all zeros) and the entire Hyrox flow looks "working" but yields a zero-length plan.

**Fix:** Same parse guard as BUG-M1, both files.

### BUG-M3 — Ad-hoc `startOfDay` shadows the `date-utils` helpers
**Files:**
- `src/lib/hyrox-archive.ts:33-37`
- `src/lib/hyrox-blocks.ts:285-300`
- `src/app/api/hyrox/today/route.ts:161-165`
- `src/app/body/hyrox/page.tsx:242-246`

Each defines its own `startOfDay(d)` using `setHours(0,0,0,0)` (local TZ). The codebase already standardized on `getLocalDay()` / `getLocalDayStr()` in `src/lib/date-utils.ts` to fix BUG-008. Mixing local-TZ start-of-day with Prisma `Date` values (stored UTC) produces off-by-one day boundaries near midnight in non-UTC zones — exactly the failure BUG-008 was meant to close.

Practical impact today is small (most math is module-internal and consistent), but it's a regression of the established convention and will bite when these dates are compared against `day` fields elsewhere.

**Fix:** Replace each local definition with imports from `src/lib/date-utils.ts`. Audit any cross-comparison against Prisma-stored `day` fields.

### BUG-M4 — Goals PATCH accepts unvalidated `type`, `subtype`, `priority`
**File:** `src/app/api/goals/[id]/route.ts:15-17`

POST validates `type` against the seven-value enum. PATCH does not — the same single loop copies `type`, `subtype`, `priority` straight through. A bad `type` breaks `goalLenses[goal.type]` lookup; a bad `subtype` breaks the new `race_hyrox` lens key resolution (`${type}_${subtype}`). `priority` accepts any value including non-numeric (Prisma will reject non-int but a string number "5" coerces and hides the bug).

**Fix:** Reuse the POST validators in PATCH. Validate `priority` is `Number.isInteger(n) && n >= 0`.

### BUG-M5 — 14-day pace-gap window uses unnormalized epoch math
**File:** `src/lib/coach-context.ts:459`

```ts
const fourteenDaysAgo = new Date(Date.now() - 14 * 24 * 3600 * 1000);
const withIntervals = recentSessions.filter(s => s.intervalsJson && new Date(s.day) >= fourteenDaysAgo);
```

No TZ normalization, so the window slides through midnight relative to the user's local "today." Sessions logged late at night on day-15 are sometimes in, sometimes out, depending on what time the coach is queried. Same family as BUG-M3.

**Fix:** Normalize to start-of-day via `getLocalDay()` then subtract 14 days.

### BUG-L3 — `maybeArchivePlan` race-day boundary uses local-TZ start-of-day
**File:** `src/lib/hyrox-archive.ts:22-24`
