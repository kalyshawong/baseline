## High

### BUG-H1 ŌĆö `/api/nutrition` has no rate limit on its Anthropic call
**File:** `src/app/api/nutrition/route.ts:65`

```ts
const estimates = await estimateMacros(text);
```

`estimateMacros()` sends up to 4 KB of user-supplied free text to Claude on every POST. The route already caps `text` length (good) but applies no request throttling. A misbehaving client (stuck retry loop, browser extension, tab left auto-refreshing) can rack up unbounded token cost ŌĆö the exact failure mode that BUG-004 closed for `/api/coach`. The `/coach` route uses an in-memory `checkRateLimit("coach")` bucket; the nutrition route should do the same.

Practical impact today is moderate (single-user app, low traffic) but the cost exposure grows linearly with any frontend bug or accidental loop. The `estimateMacros` failure mode is also silent-on-the-client: a Claude error bubbles up as a 500 from `apiError()`, but no per-call cost is logged.

**Fix:** Mirror the BUG-004 pattern from `coach/route.ts:17-26` ŌĆö keep a rate bucket keyed `"nutrition"` (or share the coach bucket since they hit the same API), reject the 11th request in a 60s window with `429`. Optionally add token-usage logging via `withAnthropicRetry`'s `label` so spikes are visible in server logs.

### BUG-H2 ŌĆö `/mind` page `Promise.all` is now an all-or-nothing read of 11 sources
**File:** `src/app/mind/page.tsx:44-97`

```ts
const [experiments, dayTags, dayReadiness, daySleep, dayStress, cyclePhase,
       latestEnv, insights, nutritionLog, lifeContextDefs, lifeContextLogs]
  = await Promise.all([ /* 11 prisma + insights() queries */ ]);
```

This diff window added the two LifeContext queries (`02ae462`, `4f8302f`) and `nutritionLog` to the fan-out. There is no surrounding try/catch ŌĆö if any one query rejects (transient DB blip, `generateInsights()` throwing on its own internal `Promise.all` of 6 more queries), the whole `/mind` page renders a 500. The downstream components are individually permissive of `null`/empty (TagTimeline, NutritionLog, LifeContextCard all handle empty states), so partial-fail should degrade to a usable page, not blank.

This is the same BUG-006 family that `buildCoachContext()` already fixed via `Promise.allSettled` + `val()` extractor. Heightened risk now because the page issues 11 + 6 = up to 17 concurrent reads ŌĆö much more surface area for one query to time out.

**Fix:** Switch to `Promise.allSettled` and extract each value defensively (treat rejection as `null`/`[]`). Same pattern as `src/lib/coach-context.ts:~880` (the existing fix). Wrapping the whole block in try/catch with a fallback render is acceptable but loses the partial-data win.

### BUG-M2 ŌĆö `/` dashboard `Promise.all` on 12 queries; swallow-all try/catch hides which one failed
**File:** `src/app/page.tsx:80-109` (plus 120-141 follow-up `Promise.all`s)

The outer dashboard query block IS wrapped in `try { ... } catch { /* DB not connected ŌĆö show empty state */ }`. So one transient failure does not 500 the page; instead it silently blanks all metric cards. That's worse than a 500 in some ways ŌĆö the user sees a "no data" dashboard and doesn't know whether they actually have no data or whether the DB hiccuped.

**Fix:** Either `Promise.allSettled` per the BUG-H2/H3 pattern (partial data is much better than zero), or at minimum log the caught error server-side so the swallowed failure is visible. The current bare `catch {}` discards `error` entirely.

### BUG-M3 ŌĆö `insights.ts` buckets `tag.timestamp` and `eatenAt` by UTC date but biometrics by local-stored `day`
**File:** `src/lib/insights.ts:204, 305, 259`

```ts
const dayStr = t.timestamp.toISOString().split("T")[0];           // line 204 (tags)
const dayStr = e.eatenAt.toISOString().split("T")[0];             // line 305 (nutrition entries)
const dayStr = l.day.toISOString().split("T")[0];                 // line 259 (nutrition logs - day-keyed, fine)
```

`ActivityTag.timestamp` and `NutritionEntry.eatenAt` are wall-clock instants (any time of day). `.toISOString().split("T")[0]` collapses them to a UTC date. For a Pacific user logging a tag at 17:00 local Tue (= 00:00 UTC Wed), the tag is bucketed under Wed ŌĆö but `DailySleep.day` and `DailyReadiness.day` are stored as UTC-midnight of the local day (per `dateStrToUTC()`), so they're bucketed correctly under Tue. The two sides go into `compareBuckets()` and end up comparing the wrong days.

Impact: insight correlations near midnight silently shift by one day. A tag logged at 5pm Tuesday gets compared against Wednesday's biometrics. This is the same BUG-008 / BUG-M3-from-prior-scan timezone family.

`LifeContextLog` is safer here because the writer (`/api/life-context/logs`) already calls `dateStrToUTC()` on the input `day` string, so its bucket matches the biometric storage convention. Only the timestamp-based sources have the skew.

**Fix:** Convert `t.timestamp` and `e.eatenAt` to local-day strings via the existing `getLocalDayStr()` style, or compute via `getLocalDayBounds(dateStr)` ŌĆö whichever matches the way biometric `day` fields were originally bucketized at write time. Document the convention in a comment near `metricConfigs`.

### BUG-M4 ŌĆö `/api/workouts` POST passes unvalidated `date` body field through `new Date()`
**File:** `src/app/api/workouts/route.ts:32-43`

```ts
const { templateName, date } = body;
const isBackfill = !!date;
const sessionDate = isBackfill
  ? new Date(date + "T00:00:00.000Z")  // no validation
  : getLocalDay();
const startedAt = isBackfill
  ? new Date(date + "T12:00:00.000Z")  // no validation
  : new Date();
```

The 2026-05-05 validation pass touched this file (added `parseIntInRange` for the GET handler) but missed the POST body's `date` field. A client passing `"2026-13-99"` or `"garbage"` gets `Invalid Date` ŌåÆ Prisma rejects with a generic 500, not a clean 400. Same BUG-M2 family as the prior-scan goals route ŌĆö closed there, regression here.

Also `templateName` is written straight to the DB with no length cap or type guard. Compare to the `goals` POST which validates `title` against `TITLE_MAX = 200`.

**Fix:** Use the existing `validateDateString` helper on `date`, validate format with the YYYY-MM-DD regex like nutrition route does (`/^\d{4}-\d{2}-\d{2}$/`). Add `validateString(templateName, "templateName", { maxLen: 200 })`.

### BUG-M5 ŌĆö Nutrition entry creates + total update are not transactional
**File:** `src/app/api/nutrition/route.ts:90-123`

```ts
for (const est of estimates) {
  await prisma.nutritionEntry.create({ ... });   // sequential creates
  addedCals += est.calories; /* etc */
}
const updated = await prisma.nutritionLog.update({
  data: { calories: { increment: addedCals }, ... },
});
```

If the loop fails mid-iteration (constraint violation, DB connection drop), some entries get created and the daily totals are never updated. The next read sees orphaned entries that don't reconcile with `NutritionLog.calories`. The downstream `MacroSummary` and TDEE math then disagree with the entry list visible in `NutritionLog` UI.

Also: the `ActivityTag.create` at the end (line 130) is outside any transaction. If it fails after entries succeed, the entries exist but the auto-tag for experiment integration doesn't ŌĆö silently breaking nutrition-tied experiment correlations.

**Fix:** Wrap the entries-loop + log-update + activity-tag-create in `prisma.$transaction(async (tx) => { ... })`. Either all four succeed or none.

### BUG-L3 ŌĆö `/api/templates` POST has thin shape validation but no numeric/length bounds
**File:** `src/app/api/templates/route.ts:25-42`

```ts
const { name, split, exercises } = body;
if (!name || !Array.isArray(exercises) || exercises.length === 0) return 400;
for (const ex of exercises) {
  if (!ex.exerciseName || !ex.targetSets || !ex.targetReps) return 400;
}
```

Checks truthiness only. `name` can be any length (no `TITLE_MAX`), `exerciseName` can be 10 KB of garbage, `targetSets`/`targetReps` accept strings ("4"), negatives, floats, or NaN ŌĆö anything that's truthy. None of these crash today because the field is JSON-stringified into a single column, but the downstream reader (`workout-logger`, `safeJsonParse`'d) will eventually do math on them and silently produce NaN.

This route was lightly touched in `2ebca6e` (swapped raw `JSON.parse` for `safeJsonParse`) but the input-shape pass didn't reach this handler.

**Fix:** Apply `validateString` on `name` (cap 200), `validateString` on each `exerciseName` (cap 200), `validateNumber` on `targetSets`/`targetReps` (integer, min 1, max 50 / 100 respectively). Reuse the `WORKOUT_SET` ranges from the sets route for consistency.
