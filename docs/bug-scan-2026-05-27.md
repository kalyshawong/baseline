## High

### BUG-H1 — `/api/workouts` POST `date` body field is still unvalidated (regression — BUG-M4 from prior scan, now elevated)
**File:** `src/app/api/workouts/route.ts:29-67`

```ts
const body = await request.json();
const { templateName, date } = body;

const isBackfill = !!date;
const sessionDate = isBackfill
  ? new Date(date + "T00:00:00.000Z")   // no validation
  : getLocalDay();

const startedAt = isBackfill
  ? new Date(date + "T12:00:00.000Z")   // no validation, second injection site
  : new Date();
```

The prior scan flagged this as **BUG-M4** (medium). The uncommitted diff *doubles down* on the pattern — `date` is now string-concatenated into **two** Date constructors, and a new UI surface (the `/body/workout/new` date picker, BUG-M1 below) makes this the user-facing entry point for the field rather than a power-user backfill API. That's why it moves up to HIGH.

Failure modes a client can trigger today by sending a crafted `date`:
1. `date: "garbage"` → both Dates become `Invalid Date` → Prisma `P2000`-class error → generic 500 via `apiError()`, not a clean 400.
2. `date: "2026-13-99"` → same outcome.
3. `date: "2099-01-01"` → no max-bound check; row created with `startedAt` 73 years in the future. The HTML `max={todayStr}` attribute on the picker is client-side only.
4. `date: "2026-05-27T<script>"` → string concatenation means anything after the date gets appended to the timestamp. `new Date("2026-05-27<script>T00:00:00.000Z")` → `Invalid Date`, but the unsanitized input flows further than it should.

Also still open from BUG-M4: `templateName` is written to the DB at line 58 with no length cap or type guard — compare `goals` POST which validates `title` against `TITLE_MAX = 200`.

**Fix:** Reach for the helpers already shipped in `2ebca6e`:

```ts
const dateErr = date !== undefined
  ? validateDateString(date, "date", { allowNull: false })
  : null;
const templateErr = validateString(templateName, "templateName", {
  maxLen: 200, allowNull: true,
});
const errors = collectErrors([dateErr, templateErr]);
if (errors) return NextResponse.json({ errors }, { status: 400 });
```

Then reject `date` strings whose parsed value is in the future (e.g., `dateStrToUTC(date) > getLocalDay()` → 400). Switch the two `new Date(date + "T...")` calls to `dateStrToUTC(date)` and a one-line `+ 12h` offset to remove the two string-concat sites.

### BUG-H2 — Backfill detection on `/body/workout/[id]` uses UTC-date strings; misfires for users west of UTC in the evening
**File:** `src/app/body/workout/[id]/page.tsx:103-106`

```ts
// Detect backfill: session date is before today (UTC day comparison)
const sessionDay = session.date.toISOString().split("T")[0];
const todayDay = new Date().toISOString().split("T")[0];
const isBackfill = sessionDay < todayDay;
```

`session.date` was stored as UTC-midnight of the user's *local* day (via `getLocalDay()` or `dateStrToUTC(date)`), so `sessionDay` correctly reads back as that local-day string. But `new Date().toISOString().split("T")[0]` is wall-clock UTC right now — for a Pacific user at 6pm local on 2026-05-27, that resolves to `"2026-05-28"` (UTC date). A session created seconds earlier via `getLocalDay()` has `sessionDay = "2026-05-27"`. The comparison `"2026-05-27" < "2026-05-28"` is **true** → the page renders in backfill mode for a workout that's happening right now → BUG-H3 cascades: rest timer is hidden and not auto-started after each set.

Severity is HIGH because this is the exact BUG-008 timezone family the codebase is supposed to be done with, and the regression silently breaks the rest-timer UX (a primary value driver of the live workout flow) for every west-of-UTC user every evening. The comment on line 103 even calls out "UTC day comparison" as if it were a feature — it should be local.

**Fix:** Use the existing helper:

```ts
import { getLocalDayStr } from "@/lib/date-utils";
const sessionDay = session.date.toISOString().split("T")[0]; // still correct since stored at UTC-midnight of local day
const todayDay = getLocalDayStr();
const isBackfill = sessionDay < todayDay;
```

Update the comment to "local-day comparison". Same fix applies to BUG-M1.

## Medium

### BUG-M1 — `/body/workout/new` date picker defaults to UTC-today, not local-today
**File:** `src/app/body/workout/new/page.tsx:32`

```ts
const todayStr = new Date().toISOString().split("T")[0];
```

Same BUG-008 family as BUG-H2 above, in a client component this time. Two user-visible consequences for west-of-UTC users in the evening:
1. The `<input type="date" max={todayStr} />` picker on line 142 lets the user select tomorrow's local date (because UTC has already rolled over), which then races through the unvalidated POST path (BUG-H1) and creates a future-dated session.
2. The "is this a backfill?" decision at line 61 (`if (workoutDate !== todayStr) payload.date = workoutDate`) misfires symmetrically — picking the actual local today sends `date` to the server as a backfill anyway.

Severity is one notch below BUG-H2 because this is a client component and `todayStr` is also re-evaluated on each render after the page becomes interactive; but the SSR/hydration mismatch and the wrong `max` bound are real.

**Fix:** Compute `todayStr` from local components, mirroring `getLocalDayStr()`:

```ts
const now = new Date();
const todayStr = `${now.getFullYear()}-${String(now.getMonth()+1).padStart(2,"0")}-${String(now.getDate()).padStart(2,"0")}`;
```

The `date-utils.ts` helper is server-only as written (no "use client"), so duplicate the 1-liner or factor a shared `lib/date-client.ts`. The duplication is fine for one site; if BUG-M3 below also lands, factor it.

### BUG-M2 — `startWorkout` fetch ignores non-OK responses; silent failure on validation rejection
**File:** `src/app/body/workout/new/page.tsx:64-73`

```ts
const res = await fetch("/api/workouts", { ... });
if (res.ok) {
  const session = await res.json();
  router.push(`/body/workout/${session.id}`);
}
// else: nothing
```

If the server returns 400 (which it will once BUG-H1 is fixed and a bad date is submitted) or 500, the click vanishes — `isPending` flips off, no error surfaces, and the user re-clicks. This is the BUG-009 family the codebase already remediated across other client components; the new file regressed because it was authored without the `setError` / inline-error pattern used by `WorkoutLogger`, `QuickTag`, etc.

**Fix:** Add an `error` state and render it under the "Choose a Template" header. Pattern (1:1 with `WorkoutLogger`):

```ts
const [error, setError] = useState<string | null>(null);
// ...
if (!res.ok) {
  const data = await res.json().catch(() => ({}));
  setError(data?.errors?.join(", ") || data?.error || "Failed to start workout");
  return;
}
```
